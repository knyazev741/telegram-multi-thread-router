**In `src/bot/routers/session.py`:**

1. Add imports at top:
   ```python
   from aiogram.types import Message, CallbackQuery, ReactionTypeEmoji
   from src.sessions.permissions import PermissionCallback, PermissionManager
   from src.config import settings
   ```

2. Add callback query handler BEFORE the existing message handlers (order matters for aiogram):
   ```python
   @session_router.callback_query(PermissionCallback.filter())
   async def handle_permission_callback(
       query: CallbackQuery,
       callback_data: PermissionCallback,
       permission_manager: PermissionManager,
   ) -> None:
   ```
   Implementation:
   - Guard: check `query.from_user and query.from_user.id != settings.owner_user_id` -> `await query.answer()` and return silently (Pitfall 5 -- defensive owner check for callback queries since OwnerAuthMiddleware only covers messages)
   - Call `resolved = permission_manager.resolve(callback_data.request_id, callback_data.action)`
   - If NOT resolved (stale): `await query.answer(text="This permission has expired", show_alert=True)` and return (PERM-08)
   - If resolved: `await query.answer()` to dismiss spinner (CRITICAL: must call answer before any other awaits -- Pitfall 1)
   - Edit the original message to remove keyboard: `if query.message: await query.message.edit_reply_markup(reply_markup=None)`
   - Append resolution label to message: map action to label {"allow": "Allowed once", "always": "Always allowed", "deny": "Denied"}, then `await query.message.edit_text(query.message.text + f"\n\n<b>Result:</b> {label}", parse_mode="HTML")` -- wrap in try/except to handle already-edited messages gracefully

**In `src/bot/dispatcher.py`:**

3. Register OwnerAuthMiddleware on `dp.callback_query.outer_middleware` too (not just `dp.message`):
   ```python
   auth_mw = OwnerAuthMiddleware(owner_id=settings.owner_user_id, group_chat_id=settings.group_chat_id)
   dp.message.outer_middleware(auth_mw)
   dp.callback_query.outer_middleware(auth_mw)
   ```
   NOTE: Check if OwnerAuthMiddleware works with CallbackQuery (it checks `event.from_user.id` and `event.chat.id`). CallbackQuery has `from_user` but chat is in `query.message.chat`. If the middleware uses `event.chat.id`, it may fail for callbacks. If so, keep the in-handler guard from step 2 above instead and skip this middleware registration. The in-handler guard is sufficient and simpler.

   DECISION: Use the in-handler guard only (step 2 above). Do NOT register OwnerAuthMiddleware for callback_query -- it was designed for Message events and CallbackQuery has different attribute paths. The defensive check in the handler is cleaner.
